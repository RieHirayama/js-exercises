## 問題 15.11-15.15 🖋

Slack や O365 の API は機密情報をヘッダーで指定する。パスやクエリでないのは何故かセキュリティの観点で記載しなさい。

## 回答
Slack や O365（Microsoft Graph）などの API では、アクセストークンなどの機密情報を URL のパスやクエリではなく、HTTP ヘッダーに指定する設計になっている。

これは、URL に含まれる情報は以下のような理由で漏洩しやすいためである。

- URL はブラウザの履歴、ブックマーク、サーバーのアクセスログなどに残りやすい

- クエリ文字列は Referer ヘッダーとして他のサイトに送信される可能性がある
  - Referer ヘッダー: リクエストがどのページから送信されたかを示す HTTP ヘッダー。URL に機密情報が含まれている場合、その情報が Referer として第三者に送信される可能性があるため、認証情報を URL に含めることはセキュリティ上好ましくない。

- プロキシやロードバランサなどの中間機器でログとして保存されることがある

一方、HTTP ヘッダーは通信の制御情報として扱われ、URL よりも露出範囲を限定しやすい。特に Authorization ヘッダーは、認証情報を安全に送信するために用意された仕組みであり、HTTPS と併用することで盗聴や漏洩のリスクを低減できる。

このため、機密性の高い情報はパスやクエリではなく、HTTP ヘッダーで指定する設計が採用されている。
